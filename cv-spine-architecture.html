1. Core Concept — Why a Spine

Problem

Every CV product (surveillance, crowd analytics, attendance, perimeter security) needs same core: detect people → track them → understand pose → identify re-entry → define spatial zones → generate events. Building this from scratch per product = massive waste.

Solution: CV Spine

Single reusable pipeline with 6 core CV modules + utility layer + adapter interface. Each product = thin config + business rules + UI, sitting on top of spine. Spine team maintains models, inference, tracking. Product teams consume events.

Spine vs Product — Clear Separation

Spine Owns (Build Once)Product Owns (Per Vertical)
Person detection model + inferenceWhat "person detected" means (customer vs intruder vs student)
Pose keypoint extractionWhat poses matter (dancing? falling? fighting? slouching?)
ReID embedding generationWhy you track someone (attendance? VIP? suspect?)
Entry/exit line crossingWhat lines mean (door? restricted zone? checkout?)
ROI zone detectionWhat zones mean (dance floor? classroom? exclusion zone?)
Face feature extractionWho is in gallery (students? employees? watchlist?)
Frame ingest + decode + distributeCamera layout, venue topology
GPU scheduling + model servingWhich models to enable per camera
Event bus (MQTT/Redis)Which events to subscribe to, alert logic
TimescaleDB schema + retentionProduct-specific aggregation queries

Derived Signals (spine computes, product interprets)

  • Fall detection: keypoint velocity spike + horizontal torso angle > 60° → pose.fall_detected
  • Dance energy: limb velocity + periodicity → pose.dance_energy: 0-100
  • Fight detection: 2+ persons close + high arm velocity + arm extension → pose.aggression_detected
  • Loitering: low movement magnitude for > N seconds → pose.loitering
  • Attention direction: head keypoints → gaze direction estimate

Pillar 3: Re-Identification (Body-based) REID

Models

ModelEmbedding DimSpeedStrength
OSNet-x1.05122ms/cropBest body-only ReID. Works without face. Clothing-based.
FastReID (BoT)20484ms/cropHigher accuracy, heavier. Good for cross-camera.
CLIP ViT-B/16 (body crop)5126ms/cropZero-shot, works across domains without fine-tuning

How It Works

1. PersonDetector emits bbox crop
2. ReID module extracts 512-dim embedding from crop
3. Embedding compared against gallery (Qdrant vector DB)
   - Cosine similarity > 0.75 → "same person"
   - Below threshold → "new person", add to gallery
4. Gallery has TTL — embeddings expire after N hours
   (or persist for enrolled users: employees, students)

Events:
  "reid.match"       → person re-identified (track_id → gallery_id)
  "reid.new_person"  → first time seen today
  "reid.cross_camera"→ same person seen on different camera

Privacy Architecture

  • Body-based ReID = no facial biometrics. Clothing/shape/gait based. GDPR-friendlier.
  • Embeddings stored in Qdrant with TTL. Not reversible to image.
  • Product layer decides gallery retention policy (schools: session-only, military: persistent)

Line-of-Regard (LoR) System

# Product defines lines in camera view:
lines:
  - id: "main-entrance"
    type: "entry_exit"           # bidirectional
    points: [[100, 500], [600, 500]]  # pixel coords
    direction_in: "up"           # crossing upward = entry

  - id: "restricted-door"
    type: "entry_only"           # one-way
    points: [[700, 200], [700, 600]]
    alert_on: "exit"             # alert if someone exits through entry-only

  - id: "checkout-line"
    type: "counting"
    points: [[0, 400], [1280, 400]]

# Spine computes:
# 1. Track centroid path history
# 2. Ray-segment intersection with each line
# 3. Direction via velocity vector dot product with line normal
# 4. Emit: line.crossed { line_id, track_id, direction: "in"|"out" }

Output Events

"line.crossed"    → { line_id, track_id, direction, timestamp }
"track.created"   → { track_id, first_bbox, camera_id }
"track.lost"      → { track_id, last_bbox, duration_seconds, path }
"dwell.exceeded"  → { track_id, zone_id, dwell_seconds }
"occupancy.count" → { camera_id, zone_id, count, delta }

4. Utility Layer

Shared tools every module uses. Part of spine, not product.

UTIL Frame Ingest Manager

  • Multi-protocol: RTSP, RTMP, HLS, USB, IP camera HTTP, file
  • Background reader thread — always latest frame, no buffer bloat
  • Auto-reconnect with exponential backoff
  • Health monitoring per camera (FPS, latency, packet loss)
  • Resolution normalization (configurable target)

UTIL GPU Scheduler

  • Multi-model batching: collect frames from N cameras → single GPU batch
  • Priority queues: real-time > analytics > background
  • Memory-aware scheduling: track per-model VRAM, prevent OOM
  • Fallback: GPU busy → queue or drop frame (configurable per product)
  • Multi-GPU support: round-robin or affinity-based

UTIL Frame Deduplication

  • Perceptual hash (pHash) per frame
  • If Hamming distance < threshold from last processed → skip inference
  • Saves 60-80% GPU on static cameras (parking lots, hallways at night)
  • Configurable sensitivity per camera/product

UTIL Anonymizer

  • Runs BEFORE any frame leaves spine (MQTT, storage, WebSocket)
  • SCRFD face detection → Gaussian blur / pixelation / solid fill
  • Body anonymization option (full silhouette replacement)
  • License plate blur (YOLO-trained or PaddleOCR detection)
  • Configurable: blur strength, region expansion, bypass for recognized gallery

UTIL Model Registry

  • Central catalog of all loaded models
  • Hot-reload: swap model version without restart
  • A/B testing: run 2 model versions, compare outputs
  • Format support: PyTorch, ONNX, TensorRT, OpenVINO
  • Version tracking + rollback

UTIL Metrics & Observability

  • Prometheus metrics: inference_latency, frames_processed, detections_count, gpu_util, queue_depth
  • OpenTelemetry tracing: frame → detection → tracking → event
  • Model drift monitoring: confidence distribution over time
  • Camera health: FPS, frame drops, reconnects
  • Per-product dashboards via Grafana

5. Adapter / Plugin System

Product Adapter Interface

from abc import ABC, abstractmethod
from typing import Dict, List, Any
from cv_spine.events import SpineEvent
from cv_spine.config import CameraConfig, ZoneConfig, LineConfig


class ProductAdapter(ABC):
    """
    Every product implements this. Spine calls these methods.
    Product never touches spine internals — only receives events and provides config.
    """

    @property
    @abstractmethod
    def product_id(self) -> str:
        """Unique product identifier: 'vibecheck', 'schoolguard', etc."""
        ...

    @abstractmethod
    def get_camera_configs(self) -> List[CameraConfig]:
        """Which cameras, what modules enabled, what params."""
        ...

    @abstractmethod
    def get_zone_configs(self) -> List[ZoneConfig]:
        """ROI zones, exclusion zones, density grids per camera."""
        ...

    @abstractmethod
    def get_line_configs(self) -> List[LineConfig]:
        """Entry/exit lines, tripwires, counting lines."""
        ...

    @abstractmethod
    def get_face_galleries(self) -> Dict[str, Any]:
        """Face recognition galleries (if face module enabled)."""
        ...

    @abstractmethod
    def on_event(self, event: SpineEvent) -> None:
        """
        Spine pushes events here. Product decides what to do:
        - VibeCheck: update vibe score
        - SchoolGuard: mark attendance
        - PerimeterAI: trigger alarm
        """
        ...

    def get_alert_rules(self) -> List[Dict]:
        """Optional: declarative alert rules. Default = handle in on_event."""
        return []

    def get_analytics_config(self) -> Dict:
        """Optional: what to persist, at what granularity."""
        return {"persist_events": True, "aggregation_interval": 60}


# ── Example: VibeCheck Adapter ──

class VibeCheckAdapter(ProductAdapter):
    product_id = "vibecheck"

    def get_camera_configs(self):
        return [
            CameraConfig(
                camera_id="cam-main",
                source="rtsp://mediamtx:8554/venue-001",
                modules={
                    "person_detector": {"model": "yolov8m", "confidence": 0.35},
                    "pose_keypoints":  {"model": "yolov8m-pose", "fps": 2.0},
                    "roi_zones":       {"enabled": True},
                    "entry_exit":      {"enabled": True},
                    "reid":            {"enabled": True, "gallery_ttl": 14400},
                    "face":            {"enabled": False},  # privacy: no face in clubs
                },
                stream_to_browser=True,
                anonymize=True,
            )
        ]

    def get_zone_configs(self):
        return [
            ZoneConfig(id="dance-floor", camera_id="cam-main", type="polygon",
                       points=[[200,300],[800,300],[900,700],[100,700]],
                       analytics={"heatmap": True, "dwell": True}),
            ZoneConfig(id="bar-area", camera_id="cam-main", type="polygon",
                       points=[[0,0],[400,0],[400,300],[0,300]]),
        ]

    def on_event(self, event):
        if event.type == "pose.dance_energy":
            self.vibe_engine.update_dance_score(event.data)
        elif event.type == "occupancy.count":
            self.vibe_engine.update_occupancy(event.data)
        elif event.type == "zone.overcrowded":
            self.alert_service.send("overcrowded", event.data)

8. Inference Infrastructure

Option A: Embedded (Current VibeCheck Style)

# Models loaded in-process. Good for single-node, 1-10 cameras.
# Simple but doesn't scale.

cv-worker (Python process)
├── YOLO model (GPU)
├── Pose model (GPU)
├── SCRFD (GPU)
├── ReID model (GPU)
└── Per-camera threads (CPU)

Pros: Simple deploy, low latency (no network hop)

9. Deployment Models

Cloud-First (SaaS)

Customer camera → RTSP over VPN → Cloud GPU
                                   ↓
                              CV Spine (K8s)
                                   ↓
                          Product API + Dashboard

Best for: VibeCheck, RetailIQ
Latency: 100-500ms (acceptable for analytics)
Cost: Per-camera subscription
Security: VPN tunnel for video feed

Edge (On-Premises)

Customer camera → Local Jetson/Server
                      ↓
                 CV Spine (Docker)
                      ↓
               Local Dashboard + Cloud Sync

Best for: HomeWatch, Military, Schools
Latency: <50ms (local inference)
Cost: Hardware one-time
Privacy: Video never leaves premises

Hybrid

Camera → Edge device (detection + anonymization)
              ↓
         Anonymized events → Cloud
              ↓
         Cloud analytics + dashboard

Best for: Enterprise with privacy requirements
Latency: Detection local (fast), analytics cloud
Privacy: Raw video stays on-prem, only metadata goes cloud

Multi-Tenant SaaS

Product A (VibeCheck)  ──┐
Product B (RetailIQ)   ──┤──→ Shared CV Spine Cluster
Product C (SchoolGuard)──┘    (Kubernetes, GPU pool)
                              ↓
                         Per-product namespaces
                         Per-product databases
                         Shared inference (Triton)

Most efficient at scale.
Spine team maintains cluster.
Product teams deploy adapters.

11. Build Sequence — What to Build First

Phase 1: Core Spine (Weeks 1-6)

1
Frame Grabber + Orchestrator — multi-camera ingest, frame routing, module lifecycle. This is foundation. Nothing works without it.
2
Person Detection module — YOLOv8m, configurable confidence/size/classes. Output contract finalized.
3
ByteTrack integration — multi-object tracking with track lifecycle events (created, updated, lost).
4
Event Bus — MQTT topics structure, Redis pub/sub, event schema (Pydantic models).
5
ProductAdapter ABC — interface definition, config schemas, on_event contract.
6
Anonymizer — SCRFD face detection + blur. Runs before any frame leaves spine.

Deliverable: Spine detects + tracks people on N cameras, publishes events. VibeCheck can be rebuilt on top as first product.

Phase 2: Spatial Intelligence (Weeks 6-10)

7
ROI Zone Manager — polygon zones, enter/exit detection, zone occupancy counting.
8
Entry/Exit Lines — line crossing detection, direction classification, occupancy counters.
9
Heatmap + Spatial Recommendations — density grid, flow analysis, dead zone detection.
10
TimescaleDB integration — event persistence, occupancy time-series, aggregation.

Deliverable: Spine understands WHERE people are, not just that they exist. RetailIQ becomes buildable.

Phase 3: Identity & Pose (Weeks 10-16)

11
Pose Keypoints module — YOLOv8m-pose, activity classifiers (fall, dance, fight).
12
Body ReID module — OSNet embeddings, Qdrant gallery, cross-camera matching.
13
Face Recognition module — SCRFD + ArcFace, gallery management, privacy controls.
14
GPU Scheduler + Triton — multi-model batching, dynamic scheduling, model registry.

Deliverable: Full spine operational. SchoolGuard, PerimeterAI, HomeWatch all buildable.

Phase 4: Scale & Polish (Weeks 16-20)

15
Frame Deduplication — pHash, skip inference on static scenes.
16
Observability — Prometheus metrics, OpenTelemetry, Grafana dashboards.
17
Kubernetes manifests — Helm charts, HPA, GPU scheduling, multi-tenant.
18
SDK — Python + JS client libraries for product teams.
19
Edge deployment — Jetson/RPi optimized builds, ONNX/TensorRT export.

Deliverable: Production-grade spine. Products deploy independently. Edge + cloud.
